Memperkenalkan RAG

Retrieval Augmented Generation - Cara Kerja AI yang Benar

1

Tidak Langsung Jawab

AI tidak langsung menjawab dari memorinya yang mungkin sudah outdated

2

Cari Informasi Relevan

Mencari dulu informasi yang relevan dari dokumen perusahaan

3

Formulasi Jawaban

Baru kemudian merumuskan jawaban berdasarkan informasi yang ditemukan

4

Sertakan Referensi

Setiap jawaban disertai referensi sumber yang bisa dicek

Cara Kerja RAG secara Detail

Arsitektur lengkap Retrieval Augmented Generation

Fase 1 - Persiapan (Offline)

Knowledge Base

Kumpulkan semua sumber pengetahuan: PDF, website, dokumen internal, manual produk, FAQ, regulasi

Chunking

Dokumen dipecah menjadi potongan-potongan kecil (chunks) yang bisa di-proses secara individual

Embedding & Indexing

Setiap chunk diubah menjadi vektor numerik dan disimpan di index untuk pencarian cepat

Fase 2 - Runtime (Saat User Bertanya)

User Query

User mengajukan pertanyaan. Query juga di-convert menjadi embedding vektor

Retrieval (Similarity Search)

Sistem mencari chunk yang paling mirip secara semantik dengan query dari index yang sudah dibangun

Augmented Prompt

Konteks yang ditemukan digabungkan dengan pertanyaan user menjadi satu prompt yang diperkaya

Fase 3 - Generasi Jawaban

LLM Generation

LLM menerima augmented prompt dan menghasilkan jawaban yang grounded pada konteks yang diberikan

Response + Citation

Jawaban dikirim ke user lengkap dengan referensi sumber, sehingga bisa diverifikasi kebenarannya

Inti RAG: LLM tidak mengandalkan parameter knowledge-nya sendiri, tapi "membaca" sumber eksternal dulu sebelum menjawab

Effeknya: Mengurangi hallucination, menjawab berdasarkan fakta terkini, dan bisa menunjukkan sumber referensi

Alur Kerja

Bagaimana RAG Bekerja

Alur Kerja RAG dalam 4 Tahapan

1

Retrieval

Cari dokumen atau bagian dokumen yang relevan dengan pertanyaan user

2

Augmentation

Gabungkan informasi yang ditemukan dengan konteks pertanyaan

3

Generation

AI formulasi jawaban natural berdasarkan informasi yang ada

4

Response

Berikan jawaban lengkap dengan referensi sumber

Proses ini terjadi dalam hitungan detik, tapi hasilnya jauh lebih akurat

Pendekatan Tradisional

Vector Database

Cara Lama Implementasi RAG

Pecah dokumen jadi chunk-chunk kecil (biasanya 500-1000 token)

Ubah setiap chunk jadi vector angka (embedding)

Simpan di database khusus (Pinecone, Weaviate, Chroma)

Cari chunk paling mirip dengan similarity search

DOKUMEN UTUH

[0.12, -0.34, 0.56, ...]

Vector Embeddings

Masalahnya: chunking menghilangkan konteks dan struktur dokumen

Problem

Kenapa Vector DB Bukan Solusi Ideal

Kita butuh pendekatan yang lebih sederhana dan lebih akurat

Kehilangan Konteks

Chunk kecil kehilangan hubungan dengan bagian lain dokumen

Setup Kompleks

Butuh manage embedding model, vector DB, chunking strategy

Maintenance Tinggi

Update dokumen berarti re-embed semua chunk

Biaya Infrastruktur

Vector DB punya biaya sendiri di luar LLM

Traceability Buruk

Susah tahu jawaban dari halaman mana

Pendekatan Baru

PageIndex

Vectorless RAG - RAG Tanpa Vector Database

Tidak Perlu Vector Database

Eliminasi kompleksitas infrastruktur vector DB

Tidak Perlu Chunking

Dokumen tetap utuh, tidak dipotong-potong

Tree Structure Hierarkis

Membangun hierarki dari dokumen untuk pemahaman lebih baik

LLM Reasoning Langsung

Reasoning langsung over tree structure

Perbandingan

Vector DB vs PageIndex

Vector Database

Cara Lama

  • Pecah dokumen jadi chunk kecil
  • Convert ke vector embedding
  • Simpan di database terpisah
  • Cari berdasarkan similarity
  • Butuh infrastructure tambahan
3 KOMPONEN DIBUTUHKAN

PageIndex

Cara Baru

  • Pertahankan struktur dokumen utuh
  • Bangun tree hierarchy
  • Reasoning langsung di cloud
  • Cari berdasarkan semantic understanding
  • Semua handled by PageIndex
